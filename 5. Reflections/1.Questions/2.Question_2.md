## Q2 — The Hypothesis That Was Wrong

> *Describe one hypothesis your team formed about the network capture that turned out to be wrong. What evidence had drawn you to it, and what evidence falsified it.*

---

### The Wrong Hypothesis

> **"The host `10.5.26.4` (VictoryPunk-DC) is the primary data exfiltration source, and `192.236.155.230` is the attacker's main data collection server receiving stolen credentials."**

**PCAP:** `2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap`

---

### What Drew Us To It

Three specific observations made this hypothesis look correct at first:

- **Byte volume** — `192.236.155.230` was the single highest-traffic external IP in the entire capture at **7 MB total**, ranking #1 in the Endpoints IPv4 view. Every other external IP was under 2 MB. That gap looked like bulk data leaving the network
- **Tx Bytes rank** — Sorted by Tx Bytes (outbound), `192.236.155.230` was also #1 — meaning it was not just receiving data, it was actively pushing the most data *to* internal hosts. We initially misread this as a data collection server pulling out what the implant had harvested
- **DC involvement** — By the time we were looking at this IP, the DC (`10.5.26.4`) was already confirmed compromised. It had begun its own POST exfiltration cycle at frame 13210. With the DC infected and `192.236.155.230` showing the highest byte totals, the two facts combined to look like: *DC is sending large volumes to this server*

The tshark output that reinforced the hypothesis:

```bash
tshark -r 2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap \
  -q -z endpoints,ipv4 2>/dev/null | sort -k3 -rn | head -5
```

```
Address           Packets  Bytes     Tx Bytes   Rx Bytes
192.236.155.230   3,214    7.1 MB    7.0 MB     98 kB
10.5.26.4         6,847    5.3 MB    1.2 MB     4.1 MB
10.5.26.132       4,902    4.8 MB    2.3 MB     2.5 MB
78.186.110.14     1,241    2.1 MB    2.0 MB     61 kB
91.83.88.122      1,089    1.9 MB    1.9 MB     58 kB
```

**7 MB from a single external IP, with 7.0 MB of that being Tx Bytes** — this looked like a server that was doing the heavy lifting of receiving stolen data.

---

### What Falsified It

When we drilled into the actual HTTP traffic involving `192.236.155.230`, the evidence collapsed the hypothesis immediately:

**Step 1 — Filter all HTTP involving this IP:**

```bash
tshark -r 2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap \
  -Y "http && ip.addr==192.236.155.230" \
  -e frame.number -e frame.time_relative \
  -e ip.src -e ip.dst \
  -e http.request.method -e http.request.uri \
  -e http.response.code -e http.content_length \
  -T fields -E header=y 2>/dev/null
```

```
frame  time_rel   src             dst              method  uri                        code  content_length
3052   390.115    10.5.26.132     192.236.155.230  GET     /images/redbutton.png      -     -
3056   390.234    192.236.155.230 10.5.26.132      -       -                          200   1,247,388
13295  783.009    10.5.26.4       192.236.155.230  GET     /ico/viodifot              -     -
13299  783.198    192.236.155.230 10.5.26.4        -       -                          200   4,823,614
```

**The direction was backwards from what we assumed.**

- Frame 3052: the *workstation* sent a GET to `192.236.155.230` — requesting `/images/redbutton.png`
- Frame 3056: `192.236.155.230` responded with **1.2 MB** back *to* the workstation
- Frame 13295: the *DC* sent a GET to `192.236.155.230` — requesting `/ico/viodifot`
- Frame 13299: `192.236.155.230` responded with **4.8 MB** back *to* the DC

**`192.236.155.230` was not receiving stolen data. It was delivering malware payloads and modules disguised as image files and icon paths.**

---

**Step 2 — Why 7 MB Tx Bytes did not mean what we thought:**

The Endpoints view shows **Tx Bytes as bytes *sent by that IP*.** We had read "Tx = outbound from the perspective of the external server sending to us." That was correct — but we had the *implication* backwards:

| What we thought | What it actually meant |
|---|---|
| High Tx from `192.236.155.230` = server receiving exfiltrated data and sending ACKs | High Tx from `192.236.155.230` = server actively pushing large payloads (malware modules) *to* the victim |
| DC is a major exfiltration source to this IP | DC is a major *download* destination *from* this IP |

---

**Step 3 — Confirm actual exfiltration destinations were different IPs:**

The real exfiltration POSTs — the ones with machine hostnames embedded in the URI — went to entirely different servers:

```bash
tshark -r 2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap \
  -Y "http.request.method==POST" \
  -e frame.number -e ip.src -e ip.dst \
  -e http.request.uri -e http.content_length \
  -T fields -E header=y 2>/dev/null
```

```
frame  ip.src         ip.dst            uri                                               content_length
2910   10.5.26.132    36.95.27.243      /rob87/DESKTOP-OG16DGY_W617601.../90             4911
3045   10.5.26.132    103.102.220.50    /rob87/DESKTOP-OG16DGY_W617601.../90             4911
13210  10.5.26.4      36.95.27.243      /tot108/VICTORYPUNK-DC_W617601.../90             4736
14613  10.5.26.4      103.102.220.50    /tot108/VICTORYPUNK-DC_W617601.../90             4736
```

`36.95.27.243` and `103.102.220.50` — **not** `192.236.155.230` — were the actual exfiltration destinations. These carried the machine-name-embedded URIs (Trickbot's signature collection pattern) and the outbound POST payloads of ~4,700–4,900 bytes each.

`192.236.155.230` never received a single POST.

---

### The Moment We Understood Why We Were Wrong

The URI path `/images/redbutton.png` should have been a signal earlier. A 1.2 MB file served at a path called `redbutton.png` is not an image — legitimate PNG files for web use are rarely over 500 kB, and this path has no HTML page context. Similarly, `/ico/viodifot` has no file extension at all.

Trickbot is documented to disguise its C2 module downloads as image and icon requests to blend into HTTP traffic that might pass a content-type inspection. The **high Tx Bytes from `192.236.155.230` were not data leaving the victim network — they were payloads arriving at it.**

The mistake was reading high external Tx Bytes as "server collecting data" rather than "server pushing malware." The correct interpretation: an external IP with high Tx Bytes and no matching POST requests *from* internal hosts = a **delivery server**, not a collection server.

---

### What Changed After Falsification

| | Wrong Hypothesis | After Falsification |
|---|---|---|
| **Role of `192.236.155.230`** | Primary exfiltration receiver | Trickbot module delivery server |
| **Direction of data flow** | Victim → this IP | This IP → victim |
| **Actual exfil destinations** | Thought it was this IP | `36.95.27.243` and `103.102.220.50` |
| **DC's role with this IP** | Sending harvested data | Downloading additional Trickbot modules |
| **7 MB Tx Bytes meaning** | Receiving 7 MB of victim data | Pushing 7 MB of malware payloads |
