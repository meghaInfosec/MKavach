<a id="entry1-response"></a>
**Response:**
```
## A.1 Source PCAP

**Source:** Malware-Traffic-Analysis.net, 2021 traffic analysis exercise
**Capture name:** 2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap
**SHA256:** fee3f9a285ab295821a7452a80cfaab25b8ccc60ad39052547ec67c8c3c9ca1e

**Why this capture qualifies:**
- Contains C2 beaconing — periodic Trickbot check-ins to its command-and-control
  infrastructure, plus a separate ~202-second-interval beacon with no SNI
  (consistent with a Cobalt Strike-style implant).
- Contains lateral movement / credential abuse indicators — SMB and svcctl
  (service-creation) traffic from the infected workstation to the domain
  controller, consistent with PsExec-style spreading.
- Contains data exfiltration — repeated HTTP POSTs from both the workstation
  and the DC carrying machine-fingerprint data back to external C2 servers.

All three qualifying behaviours are present in a single capture, giving full
coverage for A.3/A.4 without needing a second source.

**Why this is a defensible analogue for Meridian FinServe:**
- Trickbot is a malware family historically used as an initial-access tool
  against banks, credit unions, and other BFSI-sector organizations, often as
  a precursor to ransomware and data-theft operations.
- It is frequently followed by Cobalt Strike beacons, used to pivot from an
  initial workstation toward domain controllers and file shares holding
  sensitive data.
- Meridian FinServe's environment — a mid-sized financial firm with an
  internal Active Directory domain, handling customer PII and payment data —
  matches the exact victim profile this malware family targets.
- The attack progression (workstation compromise → credential harvesting →
  lateral movement → data exfiltration) mirrors a realistic "patient zero"
  scenario inside Meridian's network.

**Limitations of this capture as an analogue:**
- The capture comes from a generic exercise environment, not an actual
  financial-services network — host/domain names, IP ranges, and asset roles
  are not Meridian-specific.
- Workstream A will map observed hosts to equivalent Meridian roles (e.g.,
  "DESKTOP-OG16DGY" → branch workstation, "VICTORYPUNK-DC" → Meridian domain
  controller) rather than treating the capture as a literal Meridian log.
- Some hosts may use placeholder/lab names — wherever this affects confidence
  in an IOC, it will be explicitly flagged in A.4.
```
